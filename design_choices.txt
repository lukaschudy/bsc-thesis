DESIGN CHOICES — FRICTIONS
===========================
This document records every non-trivial design choice made while implementing
the frictions, along with the reasoning and the alternatives that were
explicitly considered. Intended for the methodology section of the thesis
and for supervisor review.


BASELINE (for reference)
------------------------
- Calvano et al. (2020) tabular Q-learning, two symmetric firms, logit demand.
- Grid size: m = 15 prices (confirmed Δ ≈ 0.848 — matches Calvano ~0.84).
- Memory length k = 1. State = (own price t−1, opponent price t−1).
- State encoding: s = 1 + (p1 − 1)·m + (p2 − 1). Shared state across agents.
- ε-greedy exploration with exponential decay; α learning rate; δ discount.
- Convergence: strategy unchanged for many consecutive episodes.


FRICTION 1 — OBSERVATION LATENCY
=================================

Core mechanism
--------------
Agent i observes opponent's price from period t − ℓ (instead of t − 1).
Own price lag is unchanged. Each agent's state becomes:
  s_i = (own price t−1, opponent price t−ℓ)
States are now PER-AGENT (they differ because each agent sees a different
opponent-price lag — agent i's "opponent" is agent j, and vice versa).

Design choices
--------------
(1) Per-agent vs shared state
    CHOSEN: per-agent state.
    REASON: with latency, agent 1 and agent 2 see different time horizons
    of each other, so the "state" that each is in is genuinely different.
    A shared state would require picking one agent's view as canonical,
    which is arbitrary.

(2) Own-price lag
    CHOSEN: keep at t−1 (unchanged from baseline).
    REASON: the friction is about opponent observation; self-observation
    is instantaneous. Delaying own-price would conflate two frictions.

(3) Price-history buffer
    CHOSEN: std::deque per agent, size ℓ+1, front = newest own price,
    back = oldest opponent observation.
    REASON: simple FIFO, matches the latency semantics exactly.

(4) Buffer initialization
    CHOSEN: fill all ℓ+1 slots with the agent's initial drawn price
    (same random draw as baseline, just repeated to pad the buffer).
    REASON: need SOME history to start. Using the drawn initial price
    is consistent with baseline's initialization. Repeating is simpler
    than drawing ℓ+1 independent initial prices, and during the first ℓ
    iterations exploration rate ε ≈ 1, so the buffer contents are
    irrelevant — agents play essentially at random anyway.

(5) Latency values tested
    CHOSEN: L ∈ {0, 1, 2, 3, 5, 10, 20}.
    REASON: L=0 is a sanity check (should match baseline). Fine
    resolution near the suspected breakpoint (0–3), then logarithmic
    spacing to check long-latency floor.

(6) Sessions per experiment
    CHOSEN: 250.
    REASON: Calvano uses 1000 but 250 gives stable Δ estimates (sampling
    SE ≈ 0.01) at 1/4 the compute cost. Will scale up for final runs.


FRICTION 2 — NOISY MONITORING
==============================

Core mechanism
--------------
Each agent observes the opponent's previous price corrupted by noise. The
agent cannot "denoise" — it uses the noisy observation as its state.
Formally:
  True opponent price index:     k_j ∈ {1, ..., m}
  Noise draw:                    ε ~ N(0, σ²)
  Observed opponent price index: k̂_j = clamp(round(k_j + ε), 1, m)
Per-agent state = (own price t−1, k̂_j t−1).

Design choices
--------------
(1) Noise distribution
    CHOSEN: Gaussian on the price INDEX, rounded to the grid, clamped.
    REASON: one-parameter (σ), smooth, locality-preserving (small noise
    shifts observation to nearby prices). Clamping handles boundary
    effects.
    ALTERNATIVE CONSIDERED: discrete flip (with prob p, observation is
    uniformly random across the grid). Logged in future_directions.txt.

(2) Noise on index vs on price
    CHOSEN: noise on the price INDEX (integer), not on the price value.
    REASON: keeps the observation on the grid by construction. Noise on
    price value would require a non-trivial projection back to grid and
    would scale with price range (messier interpretation).

(3) Noise symmetry
    CHOSEN: symmetric, zero-mean Gaussian.
    NOTE FOR SUPERVISOR: asymmetric noise (biased observation) is a
    reasonable variant — models systematic over/underestimation. Not
    implemented now; flagged in future_directions.txt.

(4) Which agents are noisy
    CHOSEN: BOTH agents observe each other with independent noise draws,
    same σ.
    NOTE FOR SUPERVISOR: one-sided noise (only agent 1 is noisy) is a
    different research question — information asymmetry rather than
    friction-as-policy. Logged in future_directions.txt.

(5) Independence of noise draws
    CHOSEN: each agent draws independent noise each period (fresh ε
    every t, every agent).
    REASON: models i.i.d. measurement error. Correlated noise would
    conflate with a common-shock interpretation, which is a different
    story.

(6) σ grid tested
    CHOSEN: σ ∈ {0, 0.5, 1, 2, 3, 5, 10}.
    REASON: σ=0 sanity check. σ=10 ≫ m=15 means observation is ~uniform
    (essentially no information). Grid is dense around the suspected
    breakpoint (0–3) and sparse in the high-noise regime.

(7) Random number generator
    CHOSEN: same RNG infrastructure as baseline (Mersenne Twister via
    existing globals). Seed management unchanged.
    REASON: consistency with baseline; reproducibility.

(8) State encoding
    CHOSEN: same encoding function as latency (per-agent states):
    s_i = 1 + (own − 1)·m + (obs_opp − 1). Agent 1 and agent 2 each
    compute their own perceived state.
    REASON: symmetric with the latency implementation, easy to verify
    correctness by re-deriving the σ=0 case from the baseline code.

(9) Sessions per experiment
    CHOSEN: 250 (same as latency).
    REASON: consistency across frictions for comparability.


COMMON IMPLEMENTATION CHOICES
==============================
- Each friction lives in its own LearningSimulation*.cpp/.hpp pair and its
  own main_*.cpp entry point. Reason: easy to debug, easy to ensure the
  baseline is not accidentally modified.
- Friction parameter (L or σ) read from a dedicated text file (A_Latency.txt
  / A_NoiseStd.txt) in the experiment folder. Reason: mirrors Calvano's
  A_InputParameters.txt pattern, so the folder is self-describing.
- Build via direct g++ call (build_*.bat) since cmake is not installed on
  this Windows machine. CMakeLists.txt is also kept in sync for later Linux
  / RUG cluster builds.


GRID-ROBUSTNESS RUNS (baseline, varying m)
===========================================
Supervisor asked where the collusion result "breaks" as the price grid
becomes finer. Runs live under Code/Grid Robustness/mX/.

(1) Session count
    CHOSEN: 250 sessions for m=50 and m=100 (m=15 and m=25 were run earlier
    at 1000 for Calvano replication). 250 is consistent with the friction
    experiments (same sampling SE regime) and keeps runtime tractable
    locally.

(2) Max episodes per session
    CHOSEN: 1000 episodes for m ≤ 50; 5000 episodes for m = 100.
    REASON: the Q-table has m² states (225 at m=15, 2500 at m=50, 10,000
    at m=100). With only 5,000 iterations per episode and an ε-greedy
    agent, each state is visited far less often when m is large, so
    convergence takes many more episodes. Calvano-style convergence
    requires the greedy strategy to be stable across 4 consecutive
    episodes — very hard to achieve at m=100 within 1000 episodes.
    5000 episodes (= 25M iterations per session at 5000 iters/ep) gives
    the m=100 runs a realistic chance to converge on a local machine.
    If even 5000 is too few we'll see it in the "not converged" count
    and move the m=100 run to the RUG cluster.

(3) m=100 as a local feasibility probe, not a final result
    Runs at m=100 on the local machine are primarily a feasibility test.
    If they converge, the Δ value is informative. If not, the run itself
    is evidence that the cluster is required.


FRICTION 3 — ASYNCHRONOUS ACTIONS (alternating moves, Maskin-Tirole)
=====================================================================

Core mechanism
--------------
Only one agent sets a new price per period. The other's price from the
previous period carries over unchanged. Roles alternate. At every t
exactly one price is "fresh" and one is "committed".

- Economic motivation: firms have staggered price-setting windows
  (shelf tags vs menu changes, lease renewals, etc.).
- Policy interpretation: regulator mandates staggered announcement
  windows so that at any time at least one firm's price is fixed.
- Literature: Maskin & Tirole (1988) dynamic Bertrand; Klein (2021,
  "Autonomous Algorithmic Collusion") studies Q-learners in this setup.
- Expected effect: collusion may be PRESERVED or even ENHANCED — the
  mover faces a known opponent price and can rationalize a cooperative
  best-response. Contrasts cleanly with the collusion-reducing effect
  of latency (cliff) and noise (smooth decline).

Interpretation B (asynchronous learning / updating at different rates)
was considered and moved to future_directions.txt as a potential
follow-on.

Design choices (CONFIRMED)
---------------------------

(1) Alternation schedule
    CHOSEN: strict, deterministic with lock-in length T.
    At any iteration t:
      - T == 0:  both agents move every period (baseline, simultaneous).
      - T >= 1:  "moving agent" = ((t-1) / T) mod 2 + 1.
                 So for T=1 agent 1 moves on odd t, agent 2 on even t.
                 For T=5 agent 1 moves for t=1..5, agent 2 for t=6..10,
                 agent 1 again for t=11..15, and so on.
    REASON: canonical Maskin-Tirole setup, matches Klein (2021), simplest
    to reason about. Random or stochastic alternation can be added later
    as a variant.

(2) Q-table updates for the non-mover
    CHOSEN: BOTH agents update their Q-table every period.
    The non-mover's "action" is its previous price (i.e. "do not change").
    This is a valid action in its action space; the observed reward from
    the resulting joint action is a genuine learning signal.
    REASON: discarding the non-mover's observation throws away real data.
    Both agents experience the outcome, both should learn.

(3) State representation
    CHOSEN: keep the baseline shared-state encoding
      s = 1 + (p1 - 1)*numPrices + (p2 - 1).
    Both agents see the same state; no per-agent state needed (unlike
    latency / noise).
    REASON: there is always a well-defined "last joint price" since both
    prices exist each period (one fresh, one committed). No information
    asymmetry to model.

(4) Exploration decay
    CHOSEN: epsilon decays every period for both agents, regardless of
    who moved.
    REASON: one iteration = one time step for both agents; non-movers
    also accumulate calendar time.

(5) Parameter knob (breakpoint grid)
    CHOSEN: lock-in length T ∈ {0, 1, 2, 5, 10, 20}.
    T=0 is the baseline sanity check, T=1 is pure Maskin-Tirole strict
    alternation, larger T probes how much commitment enhances or degrades
    coordination.
    REASON: gives a Δ-vs-T curve comparable to Δ-vs-L (latency) and
    Δ-vs-σ (noise). T has a direct policy interpretation (mandated
    commitment window length).

(6) Sessions per experiment
    CHOSEN: 250 (same as latency and noise).
    REASON: consistency for cross-friction comparison.

Pre-committed hypothesis (before running)
------------------------------------------
Δ does NOT decrease, and may INCREASE, with T. At T=1 (pure alternation)
the literature suggests Δ comparable to or slightly above baseline
(0.84). At larger T, agents are locked into prices for longer stretches,
which may lock in early-exploration non-optimal prices; Δ could plateau
or decline slowly. The shape to watch for: flat or gently increasing
then declining — NOT a cliff like latency and NOT a smooth monotone
decline like noise. A qualitatively different third shape.


COMBINED FRICTIONS — 2x2x2 FACTORIAL DESIGN
=============================================

Motivation
----------
We have three frictions running independently. The natural follow-up
question is whether their effects COMPOUND: if latency alone cuts Delta
by x% and noise alone by y%, does combining them cut by (1-(1-x)(1-y))
— the MULTIPLICATIVE prediction — or by (x+y) — the ADDITIVE prediction,
which is clearly wrong at the extremes but is the simple benchmark?

A full 3D grid of (L, sigma, T) is a compute-explosion trap. The
smartest minimal experiment is a 2-level factorial on all three
frictions: 8 corners of a cube.

Design
------
Each friction is "off" (level 0) or "on" (level 1), with the "on" level
chosen to produce a meaningful but non-saturating reduction in Delta:

    L:     0  or  2    (latency-only Delta ~0.215 at L=2, ~75% reduction)
    sigma: 0  or  1    (noise-only Delta ~0.493 at sigma=1, ~42% reduction)
    T:     0  or  1    (pure Maskin-Tirole strict alternation)

Reason for these levels: each is moderate — past the trivial regime but
NOT at the floor. That leaves room for interactions to be visible. Using
saturated levels (e.g. sigma=5 where Delta is already ~0) would collapse
combined regimes to a floor and wash out the interaction signal.

Factorial cells (Lxxx where each digit is on/off for L, sigma, T)
    F000  baseline                         L=0  sigma=0  T=0
    F100  latency only                     L=2  sigma=0  T=0
    F010  noise only                       L=0  sigma=1  T=0
    F001  async only                       L=0  sigma=0  T=1
    F110  latency + noise                  L=2  sigma=1  T=0
    F101  latency + async                  L=2  sigma=0  T=1
    F011  noise + async                    L=0  sigma=1  T=1
    F111  all three                        L=2  sigma=1  T=1

Total: 8 cells x 250 sessions = 2,000 sessions (~overnight locally).

Analysis plan
-------------
Report all 8 observed Delta values. Compute three benchmark predictions
for each combined-friction cell:

  Additive:        Delta_pred = Delta(0) - sum_of_single_friction_drops
  Multiplicative:  Delta_pred = Delta(0) * prod_of_single_friction_ratios
                                where ratio_i = Delta(friction_i)/Delta(0)
  Minimum:         Delta_pred = min over singles (most destructive wins)

Compare observed vs predicted. A forest plot or grouped bar chart is the
natural figure.

Implementation approach
------------------------
Write a UNIFIED learner (LearningSimulationCombined.cpp) that takes all
three parameters (L, sigma, T) and applies them in one pass. Single
binary, single set of folders. At (0,0,0) it must reproduce the
baseline exactly; at single-friction corners it must reproduce the
respective single-friction run (sanity check).

Order of operations inside a period (ordering matters):
  1. Each agent's perceived state = (own_true_t-1, noise(opp_true_t-1-L))
     — noise applies to the ALREADY-lagged observation.
  2. Moving agent chosen from (T, iter); non-mover pPrime = previous price.
  3. Moving agent selects action based on its perceived state.
  4. Reward from TRUE joint action (frictions affect observation, not payoff).
  5. Both agents Q-update with their own perceived state / action / reward.
  6. State buffers advance; next-period state computed with FRESH noise draw.
